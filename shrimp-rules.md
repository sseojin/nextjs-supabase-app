## 3. Code Standards & Style Guide

**Primary Reference**: `/Users/seojin/.claude/CLAUDE.md` (Global coding standards)
**Project Reference**: `/.claude/CLAUDE.md` (Project-specific standards)

### Mandatory Naming Conventions

- **Variables & Functions**: `camelCase` (e.g., `userId`, `getUserById`, `isEmailValid`)
- **Constants**: `CONSTANT_CASE` (e.g., `MAX_RETRIES`, `API_TIMEOUT`)
- **Classes & Types**: `PascalCase` (e.g., `UserService`, `UserDTO`, `AuthToken`)
- **Files**: Components use `PascalCase` (UserCard.tsx), utilities use `camelCase` (authUtils.ts)
- **Folders**: `camelCase` (authForm/, userProfile/)

### TypeScript Requirements

- **MANDATORY**: All functions must have explicit type annotations
- **MANDATORY**: Props interfaces must be defined for all components
- **PROHIBITED**: Using `any` type anywhere in the codebase
- **MANDATORY**: Database queries must return typed results using Database type

```typescript
// ✅ CORRECT
interface UserCardProps {
  userId: string;
  userName: string;
  isActive: boolean;
  onDelete?: () => void;
}

const getUserById = (id: string): Promise<User | null> => {
  return db.users.findById(id);
};

// ❌ PROHIBITED
const UserCard = (props: any) => {};
const getUserById = (id) => {};
```

### Comment Standards

- **WHY**: Explain the reasoning, not what the code does
- **Complex Logic**: Add step-by-step comments (초심자 수준의 상세 주석)
- **API Calls**: Document retry logic, timeouts, error handling
- **Conditional Logic**: Explain business rules that determine conditions

## 5. Component Implementation Standards

### Server vs Client Components

**Server Component** (Default - No "use client")
```typescript
// ✅ USE FOR: Data fetching, database access, environment variables
export default async function UserList() {
  const users = await fetchUsers();
  return <div>{users.map(u => <UserCard key={u.id} user={u} />)}</div>;
}
```

**Client Component** (With "use client")
```typescript
// ✅ USE FOR: State, event handlers, hooks (useState, useEffect)
"use client";
import { useState } from "react";

export default function UserForm() {
  const [formData, setFormData] = useState({});
  return <form>{/* ... */}</form>;
}
```

**PROHIBITED**:
- Using `useState`, `useEffect` in server components
- Importing client-only libraries in server components
- Fetching data directly in client components (use API routes or server actions)

### Props Interface Pattern

```typescript
// MUST follow this pattern for ALL components
interface [ComponentName]Props {
  // Required props first
  requiredProp: string;
  onAction: () => void;

  // Optional props after
  optionalProp?: boolean;
  className?: string;
}

export default function [ComponentName]({
  requiredProp,
  onAction,
  optionalProp = false,
  className,
}: [ComponentName]Props) {
  return <div className={className}>{/* ... */}</div>;
}
```

### Component Size & Responsibility

- **Maximum 300 lines**: Complex logic should be extracted to hooks
- **Single Responsibility**: Component handles one feature or display concern
- **Nested Components**: Extract sub-components if logic becomes complex

## 6. API Route Implementation Pattern

### Standard Route Handler Structure

```typescript
// app/api/[endpoint]/route.ts
import { createRouteHandlerClient } from "@supabase/auth-helpers-nextjs";
import { cookies } from "next/headers";
import { NextRequest } from "next/server";

export async function GET(request: NextRequest) {
  try {
    const supabase = createRouteHandlerClient({ cookies });

    // Authentication check if needed
    const { data: { user }, error: authError } = await supabase.auth.getUser();
    if (authError || !user) {
      return Response.json({ error: "인증 필요" }, { status: 401 });
    }

    // API Logic
    const { data, error } = await supabase
      .from("users")
      .select("*");

    if (error) throw error;

    return Response.json(data);
  } catch (error) {
    console.error("API 오류:", error);
    return Response.json(
      { error: "요청 실패" },
      { status: 500 }
    );
  }
}

export async function POST(request: NextRequest) {
  try {
    const body = await request.json();
    // Implementation
    return Response.json({ success: true });
  } catch (error) {
    console.error("API 오류:", error);
    return Response.json({ error: "요청 실패" }, { status: 500 });
  }
}
```

### Rules

- **Mandatory**: try-catch wrapper for all handlers
- **Mandatory**: Type all request/response data
- **Mandatory**: Check authentication before protected operations
- **Mandatory**: Return appropriate HTTP status codes (200, 400, 401, 404, 500)
- **Mandatory**: Log errors to console with context
- **Path Pattern**: Descriptive endpoint names (e.g., `/api/users`, `/api/auth/verify`)

## 7. Supabase Usage Standards

### Client Creation & Usage

**PROHIBITED DIRECT USAGE**:
```typescript
// ❌ WRONG - Don't do this
import { createBrowserClient } from "@supabase/auth-helpers-nextjs";
const supabase = createBrowserClient(...);
supabase.from("users").select("*");
```

**CORRECT PATTERN**:
```typescript
// ✅ CORRECT - Import from lib/supabase
import { createClient } from "@/lib/supabase/client";
const supabase = createClient();
```

### Query Location Rules

| Query Type | Location | Pattern |
|-----------|----------|---------|
| Data fetching | `lib/supabase/queries.ts` | Export named functions |
| Mutations | `lib/supabase/mutations.ts` | Export named functions |
| Auth operations | `lib/supabase/auth.ts` | Export named functions |
| Real-time subscriptions | Component file | Use within "use client" components |

### Database Type Safety

```typescript
// MUST import Database type from Supabase
import { Database } from "@/lib/types/database";

type UserRow = Database["public"]["Tables"]["users"]["Row"];
type UserInsert = Database["public"]["Tables"]["users"]["Insert"];

const getUsers = async (): Promise<UserRow[]> => {
  const supabase = createServerClient();
  const { data, error } = await supabase
    .from("users")
    .select("*");

  if (error) throw new Error(`Database error: ${error.message}`);
  return data;
};
```

## 9. Tailwind CSS Implementation

### Class Ordering Priority

When combining Tailwind classes, follow this order:

1. **Layout**: `flex`, `grid`, `block`, `inline-block`, `absolute`, `relative`
2. **Spacing**: `p-4`, `m-2`, `gap-4`, `space-y-2`
3. **Sizing**: `w-full`, `h-screen`, `max-w-md`
4. **Colors**: `bg-white`, `text-gray-900`, `border-red-500`
5. **Typography**: `text-lg`, `font-bold`, `font-sans`
6. **Effects**: `shadow-md`, `rounded-lg`, `opacity-50`
7. **Transforms**: `scale-110`, `rotate-45`, `translate-x-4`
8. **Transitions**: `transition`, `duration-300`, `ease-in-out`

```typescript
// ✅ CORRECT ORDER
<div className="flex flex-col gap-4 p-4 w-full bg-white text-gray-900 rounded-lg shadow-md">

// ❌ WRONG ORDER
<div className="shadow-md rounded-lg p-4 w-full text-gray-900 bg-white flex-col flex gap-4">
```

## 10. Form Handling & Validation

### React Hook Form + Zod Pattern

```typescript
// lib/validation/userSchema.ts
import { z } from 'zod';

export const userFormSchema = z.object({
  email: z.string().email('유효한 이메일을 입력하세요'),
  password: z.string().min(8, '비밀번호는 8글자 이상이어야 합니다'),
  name: z.string().min(1, '이름은 필수입니다'),
});

export type UserFormData = z.infer<typeof userFormSchema>;

// components/forms/UserForm.tsx
"use client";
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { userFormSchema, type UserFormData } from '@/lib/validation/userSchema';

export default function UserForm() {
  const { register, formState: { errors } } = useForm<UserFormData>({
    resolver: zodResolver(userFormSchema),
  });

  return (/* form JSX */);
}
```

### Rules

- **Validation Schema Location**: `lib/validation/[feature]Schema.ts`
- **Schema Naming**: `[feature]FormSchema` or `[feature]Schema`
- **PROHIBITED**: Duplicate validation logic between client and server
- **Server Validation**: Always validate on server-side API routes before database operations

## 11. Error Handling Standards

### Try-Catch Pattern

**MANDATORY for**:
- API routes
- Async functions
- Database operations
- External API calls

```typescript
// ✅ CORRECT ERROR HANDLING
try {
  const result = await fetchUserData(userId);
  return Response.json(result);
} catch (error) {
  console.error(`사용자 데이터 조회 실패 (ID: ${userId}):`, error);

  if (error instanceof ValidationError) {
    return Response.json({ error: error.message }, { status: 400 });
  }

  return Response.json({ error: '요청 실패' }, { status: 500 });
}
```

### Error Messages

- **User-facing**: Clear, helpful Korean message (e.g., "이메일이 이미 등록되어 있습니다")
- **Console logs**: Detailed English message with context (e.g., "Database query failed for user ID: 123")
- **PROHIBITED**: Silent failures or missing error handling

## 15. AI Decision-Making Guidelines

### When Creating New Files

1. **Component**: Check if similar component exists in `components/` → reuse/extend instead
2. **API Route**: Verify endpoint doesn't exist → check `app/api/`
3. **Type Definition**: Check if type exists in `lib/types/` → reuse instead
4. **Utility Function**: Check `lib/utils/` → add to existing file if related

### When Modifying Existing Code

1. **Read First**: Always read the complete file before making changes
2. **Check Dependencies**: Search for references to the function/component being modified
3. **Type Safety**: Ensure changes don't break TypeScript compilation
4. **Test Impact**: Identify all locations where changes might have effect

### When Unsure About Implementation

**Priority Order** (Reference in this order):
1. Existing similar code in the project (precedent)
2. Project standards (`/.claude/CLAUDE.md`)
3. Global standards (`/Users/seojin/.claude/CLAUDE.md`)
4. Framework documentation (Next.js, Supabase, TypeScript)

### Decision Tree: Server Component vs Client Component

```
Start with Server Component (default)
├─ Need state (useState)?
│  └─ YES → Use Client Component ("use client")
├─ Need event handler (onClick, onChange)?
│  └─ YES → Use Client Component ("use client")
├─ Need browser API (window, localStorage)?
│  └─ YES → Use Client Component ("use client")
├─ Can fetch data directly?
│  ├─ YES → Keep Server Component
│  └─ NO → Use API route + Client Component
└─ Default → Keep Server Component
```

## 16. Prohibited Actions & Anti-patterns

### 🚫 STRICTLY PROHIBITED

| Action | Reason | Alternative |
|--------|--------|-------------|
| Use `any` type | Breaks TypeScript safety | Define proper type or interface |
| Import Supabase directly in components | Creates hidden dependencies | Use lib/supabase/queries functions |
| Modify `components/ui/*` files | Auto-generated by shadcn/ui | Create wrapper components |
| Store secrets in `.env.local` committed | Security risk | Use `.env.example` only |
| Use `<img>` without alt text | Accessibility violation | Always add descriptive `alt` |
| Fetch data in client component | Exposes backend logic | Use API route or server action |
| Hardcode API URLs | Breaks across environments | Use environment variables |
| Conditional "use client" directives | Runtime issues | Place at module top level |
| Use arrow functions in API routes named exports | Next.js pattern violation | Use `export async function GET()` |
| Create global state without context | Causes prop drilling | Use React Context or external library |

### Anti-Pattern Examples

```typescript
// ❌ PROHIBITED - Using any type
const handleSubmit = (data: any) => {};

// ✅ CORRECT - Define proper type
interface FormData { email: string; password: string; }
const handleSubmit = (data: FormData) => {};

// ❌ PROHIBITED - Direct Supabase import
import { createBrowserClient } from "@supabase/auth-helpers-nextjs";
const supabase = createBrowserClient(...);

// ✅ CORRECT - Use prepared client
import { createClient } from "@/lib/supabase/client";
const supabase = createClient();

// ❌ PROHIBITED - Fetch in client component without API route
"use client";
const users = await supabase.from("users").select("*");

// ✅ CORRECT - Use API route
"use client";
const response = await fetch("/api/users");
const users = await response.json();
```

## 19. Project-Specific Warnings

### ⚠️ High Risk Areas

1. **Authentication Logic** (`app/auth/`, `lib/supabase/auth.ts`)
   - Changes affect all protected routes
   - Test thoroughly with various auth states
   - Coordinate with protected route middleware

2. **Database Schema** (Supabase migrations)
   - Breaking changes affect all queries
   - Cannot easily roll back production changes
   - Update types immediately after migration

3. **Environment Variables** (`.env.local`, `.env.example`)
   - Missing vars cause silent failures
   - Keep `.env.example` in sync with `.env.local`
   - Never commit sensitive data

4. **Type Definitions** (`lib/types/database.ts`)
   - Incorrect types cause runtime errors
   - Regenerate after Supabase schema changes
   - Test with actual database operations

### ⚠️ Common Mistakes

- Forgetting "use client" in components that use hooks
- Mixing server and client component patterns
- Hardcoding strings that should be constants
- Missing error handling in async operations
- Updating one file but forgetting related file

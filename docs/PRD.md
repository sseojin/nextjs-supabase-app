# 연인 데이트 코스 기획 앱 - PRD (Product Requirements Document)

## 📋 프로젝트 개요

### 프로젝트명
**Date Planner MVP** - 연인 데이트 코스 협력 기획 애플리케이션

### 목표
연인(2명)이 **하나의 지도에서 실시간으로 데이트 장소를 함께 선택하고 투표**하여 최종 데이트 장소를 결정할 수 있는 웹 애플리케이션 MVP 구축

### 핵심 문제 해결
**기존 문제**: 네이버 지도에서 장소를 하나씩 메시지로 공유하며 의사를 물어보는 불편함
- 메시지로 여러 장소 링크를 주고받음
- 어떤 장소에 동의하는지 명확하지 않음
- 최종 장소 결정이 복잡함

**제안 솔루션**: 앱을 통해 두 사람이 동일한 지도에서 동시에 장소를 선택하고 투표하여 최종 장소 결정
- 두 사람이 동일한 지도 위에서 실시간으로 장소 추가
- 각 장소에 대해 찬성/반대 투표
- 두 사람 모두 찬성한 장소만 최종 장소로 확정
- 데이트 일정을 캘린더에서 관리

## 🎯 MVP 1단계 범위

### 포함 기능 (MVP)
1. **랜딩 페이지** - 웹 제목 및 로그인 기능
2. **캘린더 메인 페이지** - 월별 캘린더로 프로젝트 관리
3. **날짜 기반 프로젝트 생성** - 캘린더에서 날짜 선택하여 프로젝트 생성
4. **공유 링크** - 고유한 공유 링크로 연인 초대
5. **네이버 지도 연동**
   - 지도 클릭으로 장소 추가
   - 검색으로 장소 추가
6. **실시간 마커 동기화** - A/B 사용자가 추가한 장소가 실시간으로 표시 (색상 구분)
7. **후보지 목록** - 오른쪽 섹션에 모든 후보지 표시
8. **투표 시스템** - 각 장소에 찬성/반대 투표
9. **실시간 투표 상태 동기화** - 투표 결과가 실시간으로 업데이트
10. **최종 확정 장소** - 두 사람 모두 찬성한 장소만 표시

### 제외 기능 (2단계로 연기)
- ⏸️ 타임테이블 드래그 앤 드롭
- ⏸️ PDF 내보내기
- ⏸️ 카카오톡 공유

```
연인 A                          연인 B
  |                              |
  └─→ 랜딩 페이지 접속           |
      ↓                           |
  └─→ 로그인/회원가입             |
      ↓                           |
  └─→ 캘린더 페이지 (2월)         |
      ↓                           |
  └─→ 2월 14일 클릭              |
      ↓                           |
  └─→ 모달: 프로젝트 제목 입력     |
      "발렌타인데이 데이트"        |
      ↓                           |
  └─→ 프로젝트 생성               |
      ↓                           |
  └─→ 캘린더에 빨간 배지 표시    |
      ↓                           |
  └─→ 배지 클릭 → 프로젝트 페이지 |
      ↓                           |
  └─→ 공유 링크 복사              |
      ├─→ B에게 링크 공유          |
      |                           |
      |                   B: 링크 클릭
      |                           ↓
      |                   B: 로그인 필요
      |                           ↓
      |                   B: 프로젝트 참여
      |                           ↓
      |                   B: "프로젝트 참여 완료"
      |                           ↓
      |                   B: 캘린더에 파란 배지 표시
      ↓                           ↓
   A와 B: 같은 프로젝트 페이지 접속
```

### 시나리오 2: 장소 추가 및 투표 (실시간 동기화)

```
화면 좌측: 네이버 지도          화면 우측: 후보지 목록
┌──────────────────┐          ┌──────────────┐
│   지도 (서울)     │          │ 장소 1: 강남역 카페
│   [빨간 마커]     │          │    👍 1/2   │
│                  │          │              │
│                  │          │ 장소 2: 홍대 식당
│                  │          │    투표 안함  │
└──────────────────┘          └──────────────┘

A: "강남역" 클릭 → 주소 자동 입력 → "카페" 카테고리 → 추가
↓ (실시간 동기화)
B 화면에 즉시 빨간 마커 표시

A: "강남역 카페" 찬성 투표 → 1/2 표시
↓ (실시간 동기화)
B 화면에 즉시 1/2 표시

B: "강남역 카페" 찬성 투표 → 2/2 확정 (초록 마커로 변경)
↓ (실시간 동기화)
A 화면에 즉시 2/2 확정 표시
```

## 🗄️ 데이터 구조

### 1. Projects (프로젝트)
```typescript
{
  id: UUID,
  title: string,           // 프로젝트 제목 (예: "발렌타인데이 데이트")
  date: Date,              // 데이트 예정 날짜 (예: 2024-02-14)
  creator_id: UUID,        // 생성자 사용자 ID
  share_link: string,      // 공유 링크 (nanoid, 예: dF3kL9xQ2pR)
  status: 'active' | 'archived' | 'completed',
  created_at: DateTime,
  updated_at: DateTime
}
```

### 2. Project Members (프로젝트 멤버)
```typescript
{
  id: UUID,
  project_id: UUID,
  user_id: UUID,
  role: 'creator' | 'member',    // creator는 빨간색, member는 파란색
  display_color: string,          // #FF0000 (creator) 또는 #0000FF (member)
  joined_at: DateTime
}
```

### 3. Locations (장소 후보)
```typescript
{
  id: UUID,
  project_id: UUID,
  added_by: UUID,          // 추가한 사용자 (creator or member)
  name: string,            // 장소명 (예: "강남역 카페")
  address: string,         // 주소
  latitude: number,        // 위도
  longitude: number,       // 경도
  naver_place_id?: string, // 네이버 지도 장소 ID
  category?: string,       // 카테고리 (카페, 식당, 영화관 등)
  memo?: string,          // 사용자 메모
  status: 'pending' | 'approved' | 'rejected',  // 투표 상태
  created_at: DateTime
}
```

### 4. Votes (투표)
```typescript
{
  id: UUID,
  location_id: UUID,
  user_id: UUID,
  vote_type: 'approve' | 'reject',
  created_at: DateTime,
  updated_at: DateTime
}
```

## 🎨 UI/UX 설계

### 페이지 구조

#### 1. 랜딩 페이지 (`/`)
- **목적**: 비로그인 사용자에게 앱 소개
- **요소**:
  - 웹 제목 및 로고
  - 앱 설명 (한두 줄)
  - "로그인" 버튼
  - "회원가입" 버튼

#### 2. 캘린더 메인 페이지 (`/dashboard`)
- **목적**: 로그인 후 프로젝트를 캘린더로 관리
- **요소**:
  - 월별 캘린더 (7x6 그리드)
  - 이전/다음 달 버튼
  - 월 표시 (예: 2024년 2월)
  - 날짜 셀:
    - 날짜 숫자 (큰 폰트)
    - 프로젝트 배지 (배지 스타일)
      - 생성자: 빨간색 배지 (#FF0000)
      - 참여: 파란색 배지 (#0000FF)
      - 최대 2개 배지, 이상은 "+N"
- **인터랙션**:
  - 빈 날짜 클릭 → 프로젝트 생성 모달
  - 프로젝트 배지 클릭 → 프로젝트 상세 페이지

#### 3. 프로젝트 생성 모달
- **목적**: 날짜를 선택하여 프로젝트 생성
- **요소**:
  - 제목: "프로젝트 생성"
  - 날짜 표시 (읽기 전용, 예: 2024년 2월 14일)
  - 프로젝트 제목 입력 필드
  - "생성" 및 "취소" 버튼

#### 4. 프로젝트 상세 페이지 (`/projects/[projectId]`)
- **레이아웃**: 화면을 좌/우 두 부분으로 나눔
- **좌측**: 네이버 지도
  - 초기 중심: 서울 시청 (37.5665, 126.9780)
  - 마커 표시 (색상: 빨강/파랑)
  - 확정 장소는 초록색 마커
  - 줌 컨트롤, 지도 타입 선택
  - 클릭으로 마커 추가 가능
- **우측**: 후보지 목록 및 투표
  - **상단**: 장소 검색 입력
  - **중간**: 후보지 목록
    - 각 항목: 장소명, 주소, 투표 버튼 (👍/👎), 투표 상태 (1/2, 2/2 확정 등)
  - **하단**: 최종 확정 장소 섹션

#### 5. 공유 링크 페이지 (`/join/[shareLink]`)
- **목적**: 공유 링크를 통한 프로젝트 참여
- **흐름**:
  1. 링크 클릭 → 로그인 여부 확인
  2. 미로그인 시 → 로그인 페이지로 리다이렉트
  3. 로그인 후 → 프로젝트 참여 요청
  4. 성공 → 프로젝트 상세 페이지로 이동

## 🔄 실시간 동기화 (Realtime)

### Supabase Realtime 구독 대상
1. **Locations 테이블**
   - INSERT: 새 마커 즉시 표시
   - UPDATE: 마커 정보 업데이트 (장소 이름, 상태 등)
   - DELETE: 마커 제거

2. **Votes 테이블**
   - INSERT/UPDATE: 투표 상태 즉시 업데이트
   - Trigger: votes 변경 시 location status 자동 업데이트

### 동기화 흐름
```
User A 액션                      Supabase                    User B 화면
─────────────                    ────────                    ────────────
지도 클릭
  ↓
후보지 추가
  ├─→ POST /locations ─────→ locations 테이블에 INSERT
                               ↓
                            Realtime 이벤트 발생
                               ↓
                          B의 구독 채널에 전달 ─→ 즉시 마커 표시
```

## 🔐 보안

### 인증 및 권한
- **기존 Supabase Auth 활용**: 이메일/비밀번호, Google OAuth 지원
- **세션 관리**: 미들웨어 기반 자동 갱신

### Row Level Security (RLS)
- `projects`: 프로젝트 멤버만 읽기/쓰기 가능
- `project_members`: 프로젝트 멤버만 읽기 가능
- `locations`: 프로젝트 멤버만 접근 가능
- `votes`: 프로젝트 멤버만 투표 가능

### 공유 링크 보안
- nanoid로 생성한 고유한 공유 링크
- 추측 불가능한 형식 (예: `dF3kL9xQ2pR`)
- 링크를 아는 사람만 참여 가능

### API 키 보호
- 네이버 API 키는 환경 변수로 관리
- Place Search는 서버 프록시를 통해서만 호출
- `NEXT_PUBLIC_` 접두사는 클라이언트 노출 항목만 사용

## 📊 역투표 로직

### 투표 상태 자동 업데이트

```typescript
// 투표가 완료되면 자동으로 location 상태 업데이트 (트리거)

IF 둘 다 찬성 THEN
  location.status = 'approved'    // 초록 마커, 확정
ELSE IF 한 명이라도 반대 THEN
  location.status = 'rejected'    // 회색 마커, 거부
ELSE
  location.status = 'pending'     // 회색 마커, 투표 진행 중
```

### UI 표시
```
상태별 UI 표현:

pending (0/2):    투표 안함
pending (1/2):    👍 1/2 (A만 찬성)
rejected (1명):   👎 거부됨
approved (2/2):   ✅ 2/2 확정 (초록 마커)
```

## ✅ 검증 시나리오

### 시나리오 1: 기본 플로우
```
1. A가 랜딩 페이지에서 로그인
2. A가 캘린더에서 2월 14일 클릭
3. A가 "발렌타인데이 데이트" 프로젝트 생성
4. 캘린더에 빨간 배지 표시 확인
5. A가 배지 클릭하여 프로젝트 페이지 진입
6. A가 공유 링크 복사
7. B가 링크 클릭 → 로그인 → 참여
8. 두 사람 모두 프로젝트 페이지 접속 확인
```

### 시나리오 2: 장소 추가 (실시간 동기화)
```
1. A가 지도에서 "강남역" 클릭
2. 주소 자동 입력 확인 (Reverse Geocoding)
3. A가 "카페" 카테고리 선택 후 추가
4. A 화면에 빨간 마커 표시 확인
5. B 화면에 즉시 빨간 마커 표시 확인 (새로고침 없이)
```

### 시나리오 3: 투표 (실시간 동기화)
```
1. A가 "강남역 카페" 찬성 투표
2. A 화면에 "1/2" 표시 확인
3. B 화면에 즉시 "1/2" 표시 확인
4. B가 "강남역 카페" 찬성 투표
5. 두 화면 모두 "2/2 확정" 표시 확인
6. 마커 색상이 초록색으로 변경 확인
```

### 시나리오 4: 에러 처리
```
1. A가 로그아웃 상태에서 프로젝트 접근
   → /auth/login으로 리다이렉트
2. C(제3자)가 공유 링크로 참여 시도
   → "프로젝트가 가득 찼습니다" 에러 메시지
3. A가 네트워크 오프라인 상태에서 장소 추가
   → "네트워크 오류" 토스트 알림
```

## 🎯 성공 기준

### MVP 완성 기준
- [ ] 두 명의 사용자가 동시에 접속하여 지도에서 실시간 동기화 작동
- [ ] 장소 추가 시 1초 이내 실시간 반영
- [ ] 투표 상태 1초 이내 실시간 반영
- [ ] 캘린더에서 프로젝트 배지가 올바르게 표시
- [ ] 공유 링크로 사용자 참여 성공
- [ ] 모든 에러 상황에서 적절한 에러 메시지 표시

### 성능 목표
- 페이지 로딩 시간: 2초 이내
- 지도 마커 렌더링: 500ms 이내
- Realtime 동기화 지연: 1초 이내

### 사용성 목표
- 직관적인 UI/UX (사용자 교육 불필요)
- 모바일 반응형 디자인
- 네트워크 오류 시 적절한 재시도 로직
